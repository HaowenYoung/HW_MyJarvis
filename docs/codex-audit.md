# Codex 跨模型审计协议

## Overview

借鉴 ARIS (Auto-claude-code-research-in-sleep) 的 auto-review-loop 设计：Claude 做判断，Codex (GPT) 审判断。跨模型审计打破 self-play 盲区——同一个模型审自己的产出有可预测的盲点，不同模型的盲点不重叠。

**审的是系统自身的判断，不是研究产出。**

## 设计原则

1. **对抗性而非验证性**: Codex 的价值在于发现 Claude 没想到的问题，不是确认 Claude 的结论
2. **审计是增强，不是门槛**: Codex 挂了 → 跳过审计，正常走原流程（degrade, don't block）
3. **呈现分歧而非强制共识**: 两个模型意见不同时，把分歧呈现给用户，不自动取舍
4. **只审高影响判断**: 不审机械操作（ingest、格式化），只审涉及归纳、推理、评估的判断

## 审计点

### 1. Distill rule candidates（每次 distill 触发）

**触发时机**: Distill subagent 产出 rule/policy/trait candidates 后，呈现给用户前。

**Codex 收到的 context**:
```
- 待审的 rule candidate 全文（Statement, Rationale, Sources, Exceptions）
- 该 candidate 的源 patterns（wiki/memory/ 中的原始 pattern 页面）
- 同 domain 的现有 active rules（避免重复/矛盾）
- 如果是 L3→L4: 被归纳的多条 L3 rules 全文
```

**Codex 审计维度**:
1. 归纳逻辑: evidence 是否真的支持 statement？有没有逻辑跳跃？
2. 过拟合风险: 是否 overfitting 到少数样本？样本多样性够吗？
3. 反例检查: 有没有明显的反例被忽略？Exceptions 是否完整？
4. 跨域归纳（L3→L4）: 跨域共性是否真实存在，还是表面相似实质不同？
5. 与现有规则的关系: 有没有跟已有 rule 矛盾？有没有其实是已有 rule 的特例？

**Codex prompt 模板**:
```
You are reviewing rule candidates generated by an AI system that manages a personal knowledge base.
Your job is adversarial review — find problems the generating model missed.

## Rule candidate
{candidate_full_text}

## Source evidence
{source_patterns}

## Existing rules in this domain
{existing_rules}

Review each candidate on these dimensions:
1. Does the evidence actually support the statement? Any logical leaps?
2. Is this overfitting to a small sample? Would it generalize?
3. Are there obvious counterexamples the Exceptions section missed?
4. Does this contradict or duplicate any existing rule?
5. (For L4 only) Is the cross-domain commonality genuine or superficial?

For each candidate, output:
- AGREE: if the rule is well-founded
- CONCERN [severity: low/medium/high]: specific issue + suggested fix
Be concise. Focus on what's wrong, not what's right.
```

**结果合并**: CC 把 Codex 意见附加到每条 candidate 下方，一起呈现给用户：
```
### Rule candidate: <title>
<Statement, Rationale, Sources, Exceptions>

**Codex audit**: AGREE / CONCERN [severity] — <具体意见>
```

### 2. Self-modification impact analysis（每次 self-modification 触发）

**触发时机**: CC 完成 Step 3 (Impact analysis) 后，进入 Step 4 (Propose) 前。

**Codex 收到的 context**:
```
- 用户的修改需求（原文）
- CC 的 impact analysis 结果
- CC 的方案草稿
- CLAUDE.md constraints 段落
- 被修改的 docs/ 文件的当前内容
```

**Codex 审计维度**:
1. 遗漏检查: impact analysis 有没有漏掉的跨模块影响？
2. 约束违反: 方案是否违反 CLAUDE.md 的任何 constraint？
3. 简化机会: 有没有更简单的实现方式达到同样目的？
4. 副作用: 修改后的交互效果——会不会让其他操作变慢/变复杂？

**Codex prompt 模板**:
```
You are reviewing a proposed system modification to a personal AI agent system.
Your job is to find what the proposer missed.

## User request
{user_request}

## Proposed modification
{proposal_draft}

## Impact analysis
{impact_analysis}

## System constraints (must not violate)
{constraints_from_claude_md}

Review:
1. Any cross-module impacts the analysis missed?
2. Any constraint violations?
3. Is there a simpler way to achieve the same goal?
4. Any unintended side effects on existing operations?

Output:
- LGTM: if analysis is thorough and proposal is sound
- MISSED: <specific gap> + suggestion
- VIOLATION: <which constraint> + how
- SIMPLIFY: <alternative approach>
```

**结果合并**: CC 把 Codex 审计意见合并到 Step 4 提案中，标注 "[Codex 审计]" 段落。

### 3. Lint 双模型交叉检查（每次 lint 触发）

**触发时机**: bash 确定性检查完成后，跟 Sonnet 一致性检查并行。

**流程**:
```
bash 确定性检查 → 结果 A
Sonnet 一致性检查 → 结果 B  ┐
Codex 独立一致性检查 → 结果 C ┘ 并行
→ diff B vs C:
  - 共识项 → 高置信（直接报告）
  - 仅 Sonnet 发现 → 标注"待确认 (Claude-only)"
  - 仅 Codex 发现 → 标注"待确认 (Codex-only)"
```

**Codex 收到的 context**:
```
- bash 确定性检查的结果（作为已确认的基础）
- wiki/rules/ 全量规则（用于 pairwise 一致性检查）
- wiki/memory/ 近期 patterns
- projects/*/wiki/INDEX.md（用于 project divergence）
```

**Codex 审计维度**（与 Sonnet lint 相同）:
1. L3 规则 pairwise 一致性（域内矛盾）
2. wiki/ 声明 vs raw/ 数据的时效性
3. 跨模块 cross-ref 完整性
4. 规则 vs 实际行为的偏差信号

**结果合并**: 三份结果（bash + Sonnet + Codex）合并为分级报告，分歧项高亮。

### 4. Experiment design review（项目 session 中自动触发）

**触发时机**: CC 在项目 repo 里完成以下任一设计决策后，自动触发：
1. Metric 定义或重新设计（如 CPP scope 怎么定、FLP 干扰项生成策略）
2. 实验参数/配置选择（如 item 格式、模型列表、评估 pass criteria）
3. 评估方法论设计（如 chain consistency 计算方式、统计检验选择）
4. 数据处理 pipeline 设计（如 distractor 生成逻辑、oracle 生成策略）
5. 统计分析方法选择（如 effect size metric、significance threshold）
6. Claim 表述（如 "ER 是 CR 的非线性高阶扩展"）

**不触发的场景**: 纯实现（写代码跑通测试）、格式修改、文档更新。只审**设计判断**。

**Codex 收到的 context**:
```
- 设计决策：做了什么决定
- 理由：为什么这么决定
- 实现细节：具体怎么做的（代码片段或伪代码）
- 已考虑的替代方案：考虑过什么别的方案，为什么没选
- 项目背景：项目目标、当前阶段、相关约束
```

**Codex 审计维度**:
1. **方法论合理性**: 这个设计是否真的测量了你想测量的东西？
2. **混淆变量**: 有没有未控制的变量可能影响结果解读？
3. **假设检查**: 设计背后的隐含假设是否成立？（如 root_cause_region 是否真是"上游控制链"）
4. **更简单的替代**: 有没有更简单、更直接的方式达到同样目的？
5. **边界情况**: 设计在极端 case 下会不会 break？
6. **可解释性**: 如果实验结果 positive/negative，你能归因到设计的哪个部分？

**Codex prompt 模板**:
```
You are reviewing an experiment design decision made during a research project.
Your job is adversarial: find methodological issues, confounds, and blind spots.

## Design decision
{what_was_decided}

## Rationale
{why_this_choice}

## Implementation
{how_implemented}

## Alternatives considered
{alternatives_and_why_rejected}

## Project context
{project_goals_and_stage}

Review on these dimensions:
1. Does this actually measure what the researcher intends? Any construct validity issues?
2. Are there confounding variables that could explain the results either way?
3. What implicit assumptions does this design make? Are they justified?
4. Is there a simpler or more direct approach?
5. What edge cases or failure modes could produce misleading results?
6. If the experiment succeeds/fails, can the result be cleanly attributed?

Output for each concern:
- LGTM: if design is sound on this dimension
- CONCERN [severity: low/medium/high]: specific issue + suggested fix
Be concise. Focus on what could go wrong, not what's fine.
```

**结果处理**:

CC 收到 Codex 意见后，呈现给用户：
```markdown
## Codex design review: <决策名>

<每条 concern 一行>
- [high] <问题>. Suggestion: <建议>
- [medium] <问题>. Suggestion: <建议>
- LGTM on: <通过的维度>

你对每条意见：acc / rej / edit？
```

用户回复后：
- `acc` 的 concern → CC 调整设计/实现，记录到 decisions.md
- `rej` 的 concern → 记录 rejection 原因到 decisions.md（将来可审计为什么 rej）
- `edit` → CC 按用户修改重新设计

**与 §1-3 的区别**:
- §1-3 审的是 MyJarvis 系统自身的判断（meta-level）
- §4 审的是 CC 在项目里做的研究设计判断（object-level）
- §4 触发频率更高（一个 session 可能触发多次）
- §4 更交互：用户逐条 acc/rej，CC 可能需要多轮调整

### 5. Literature-grounded review（§4 novelty/related work 场景触发）

**触发时机**: §4 实验设计 review 中涉及 novelty claim 或 related work 完备性时。也可由用户显式要求（"review 时查一下相关论文"）。

**不触发**: §1 规则审计、§2 self-mod（设计模式应查 repo/docs，不适合论文搜索）、§3 lint、§4 纯方法论审计（不涉及 novelty 比较）。

**流程**:

```
Step A: CC 判断本次审计是否涉及 novelty/related work
        → 不涉及：跳过，走正常 §4 流程
        → 涉及：继续

Step B: 前置搜索（Pre-injection）
        CC 从审计目标提取 2-3 组关键词
        → /arxiv 搜索（timeout 30s）
        → /semantic-scholar 搜索（timeout 30s）
        → 合并去重，按相关性筛选 top 5 篇
        → 每篇提取：title, authors, year, venue, abstract, citation count

Step C: 注入 Codex prompt
        在 §4 prompt 的 ## Project context 后增加：
        ## Relevant literature (verified via arxiv/semantic-scholar)
        {top 5 papers, each with title/year/venue/abstract/citation count}
        Note: These papers were retrieved by automated search.
              You may cite additional papers you know of,
              but mark them as [unverified] so we can check.

Step D: Codex review（正常 §4 流程）

Step E: 后置验证（Post-verification）
        CC 提取 Codex response 中引用的论文（不在 Step B 结果中的）
        → 用 /arxiv 或 /semantic-scholar 逐一验证
        → 标记状态：✅ verified / ❌ not found / ⚠️ exists but description inaccurate
        → 后置验证不阻塞报告生成

Step F: 最终报告
        Codex 审计意见 + 文献验证状态一起呈现给用户
```

**Codex prompt 模板变更**（在 §4 模板基础上增加）:
```
## Relevant literature (verified via arxiv/semantic-scholar)
{for each paper}
- [{title}]({url}) ({year}, {venue}) — cited {n} times
  Abstract: {abstract_first_200_chars}
{end for}

When assessing novelty or related work, prioritize these verified papers.
If you cite additional papers, mark them as [unverified].
```

**Context 组装原则**:
- 关键词从审计目标提取，不超过 3 组查询
- 注入不超过 5 篇核心论文（避免淹没 Codex 的审计指令）
- 每篇只注入 title/year/venue/abstract/citations，不注入全文

**审计路径论文不自动入库 raw/papers/**。如果用户认为某篇值得收录，走正常的文献调研回流路径（见 `docs/aris-integration.md` §1）。

**降级**:
| 场景 | 行为 |
|------|------|
| /arxiv 不可用 | 只用 /semantic-scholar，反之亦然 |
| 两个都不可用 | 跳过文献注入，Codex 用自带 web search，报告标注"文献未验证" |
| 单工具 timeout（>30s） | 跳过该工具，用另一个的结果 |
| 搜索结果为空 | 正常继续，Codex prompt 注明"未找到直接相关文献" |
| 后置验证发现引用不存在 | 报告中标注 ⚠️，提醒用户谨慎使用 |

## 降级策略

| 场景 | 行为 |
|------|------|
| Codex MCP 不可用 | 跳过审计，正常走原流程 |
| Codex 超时（>120s） | 跳过审计，在报告中标注"审计未完成" |
| Codex 返回无法解析的结果 | 把原始结果附给用户，不自动处理 |

**关键约束**: 审计失败永远不阻塞主流程。

## Token 成本考虑

- Distill: 低频（周/月），每次审计 ~2K input tokens + ~500 output。可忽略。
- Self-modification: 极低频，每次审计 ~5K input + ~1K output。可忽略。
- Lint: 低频（每周），Codex 一致性检查 ~10K input + ~2K output。可接受。
- Experiment design review: 中频（每个实验 session 1-5 次），~3K input + ~1K output per review。单 session 最多 ~10K output tokens。可接受。

## 与 ARIS 的区别

| | ARIS | MyJarvis |
|---|------|---------|
| 审什么 | 研究产出（论文、实验、claim） | 系统自身的判断（§1-3）+ 研究设计判断（§4） |
| 迭代 | 多轮（2-4 轮直到 pass） | 单轮（审计意见呈现给用户，用户决策） |
| 自动修复 | Claude 读审稿意见 → 自动修复 → 再审 | 不自动修复，分歧呈现给用户 |
| 通过标准 | Score ≥ 6/10 + verdict "ready" | 无自动通过标准，用户是最终裁判 |

设计选择是**单轮 + 用户决策**，因为 MyJarvis 的判断（规则、系统修改）直接影响用户的个人知识系统，不应该两个 AI 自己迭代到"共识"就定了。
